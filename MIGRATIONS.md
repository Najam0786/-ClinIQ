# ClinIQ — Database Migrations (Alembic)

ClinIQ uses [Alembic](https://alembic.sqlalchemy.org/) for schema versioning on PostgreSQL.
SQLite (local dev) continues to use `create_all` for zero-config startup.

---

## Quick Reference

| Command | Purpose |
|---|---|
| `alembic upgrade head` | Apply all pending migrations (run on deploy) |
| `alembic current` | Show current DB revision |
| `alembic history` | List all revisions in order |
| `alembic downgrade -1` | Roll back one revision |
| `alembic downgrade base` | Roll back all (⚠️ data loss) |
| `alembic stamp head` | Mark DB as up-to-date without running migrations |

---

## Local Dev (SQLite)

No migration step needed — the API auto-creates tables on startup via `create_all`.

```bash
uvicorn api.main:app --reload --port 8000
```

---

## Production (PostgreSQL)

Set `DATABASE_URL` before running:

```bash
export DATABASE_URL=postgresql://cliniq_user:password@localhost:5432/cliniq_db

# Apply all pending migrations
alembic upgrade head

# Then start the API
uvicorn api.main:app --host 0.0.0.0 --port 8000 --workers 2
```

Docker Compose handles this automatically via the `CMD` in `Dockerfile.api`:
```
alembic upgrade head && uvicorn api.main:app ...
```

---

## Creating a New Migration

After changing `db/models.py`:

```bash
# 1. Auto-generate a migration (review before applying!)
alembic revision --autogenerate -m "add_patient_notes_column"

# 2. Review alembic/versions/<rev>_add_patient_notes_column.py

# 3. Apply
alembic upgrade head
```

> **Always review autogenerated migrations** — Alembic can miss custom types,
> partial indexes, or complex constraints.

---

## Existing DB Without Alembic History

If you have a DB with the correct schema but no `alembic_version` table
(e.g. created by an older `create_all`):

```bash
alembic stamp a62d1d81b6a4   # mark as initial schema — no DDL changes run
```
